openHAB - Empowering the smart home

Design study - Goals

Some design goals emerged

This interface is build with some design principles in mind:

  • A newby friendly presentation
  • Covering all openHAB functionality (including Item Tags and Automation Rules)
  • Modern, maintainable technology stack for easy extensions.

A newby friendly presentation

  • A features tour and OH concepts explanation.
  • A guided tutorial / interactive exercises on how to begin with openHAB.
  • Contextual help texts on literally every page.
  • Everything annotatable (Things, Items, Rules, Schedule). You sometimes just need to leave a note on an Item or a Thing. Like comments in a text file.

… but expert friendly

  • Everything in on interface. Logs, OSGI bundle status, service status, upgrade button.
  • Long time stability service warnings and graphs in a central place.
  • Efficient multi item / batch editing features via textual view mode and multi-item selection.

Modern, maintainable technology stack

This new approch uses Web components (v1), static html pages, progressive enhancement
and progressive webapps (PWA) technologies like advanced caching. This means: No intransparent magic happens during the build step!

Web components

Web components are a standard web technology and allow to define own html tags like “oh-binding-doc”. No framework is required and at the same time every framework is compatible.

Such an “oh-binding-doc” custom component for example would load and display the documentation for a given binding:

<html>
  <head>
    <!-- include the component -->
    <script type="module" src="js/oh-binding-doc.js"></script>
  </head>
  <body>
    <oh-binding-doc binding="mqtt" collapsable collapsed></oh-binding-doc>
  </body>
</html>

Static Html

Static Html pages (/index.html, /items.html, /things.html) allow to only load the exact subset of necessary libraries and style sheets per page. Memory consumption is reduced, because of a small DOM. This is especially useful, because the VS-Code editor for example weights 7 MB, and is only loaded on pages that require the editor.

Style

There is a modern responsive css grid layout in place, adopting to all kind of form factors. No javascript is involved for styling at all, not for menus, not for dialogs.

Decoupled app state

A Model-View-Adapter (MVA) concept is in place to carefully decouple the model (openHAB REST interface, cached by an Indexed db) and the views from each other.

This is already put a use to, by emulating REST endpoints for not yet implemented core functionality, like a real-time logging service or a persistence configuration endpoint. Or by working around glitches in the openHAB REST API.
